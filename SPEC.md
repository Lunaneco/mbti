## 6. 診断ロジック

### 6.1 4軸

| 軸 | 左 | 右 | 意味 |
| --- | --- | --- | --- |
| エネルギー | `I` 内向 | `E` 外向 | 回復・着想の向き |
| 情報収集 | `S` 感覚 | `N` 直観 | 現実の細部か、可能性の全体像か |
| 判断 | `T` 思考 | `F` 感情 | 論理整合か、関係性/価値観か |
| 生活態度 | `J` 判断 | `P` 知覚 | 計画して進めるか、変化に開くか |

### 6.2 スコア方式

各質問は1つの軸に対応する。選択肢A/Bのどちらかに `+1` する。

例:

```ts
type Axis = "EI" | "SN" | "TF" | "JP";
type Letter = "E" | "I" | "S" | "N" | "T" | "F" | "J" | "P";

type Question = {
  id: string;
  axis: Axis;
  text: string;
  options: {
    label: string;
    letter: Letter;
  }[];
};
```

初期スコア:

```ts
const scores = {
  E: 0,
  I: 0,
  S: 0,
  N: 0,
  T: 0,
  F: 0,
  J: 0,
  P: 0,
};
```

最終タイプ:

```ts
const type =
  (scores.E >= scores.I ? "E" : "I") +
  (scores.N >= scores.S ? "N" : "S") +
  (scores.T >= scores.F ? "T" : "F") +
  (scores.J >= scores.P ? "J" : "P");
```

同点の場合:

- MVPでは上記のように右側/能動側へ寄せるルールを固定してもよい
- 推奨はタイブレイク質問を表示する方式
- タイブレイク未実装時は結果画面に `この軸は拮抗しています` と表示する

### 6.3 確信度

各軸4問なので、差分は `0` から `4`。

```ts
const confidence = 50 + Math.abs(leftScore - rightScore) * 12.5;
```

表示例:

- `E 75% / I 25%`
- `N 100% / S 0%`

表現は硬くしすぎず、結果画面では以下のように言い換える。

- `外へ星を放つ力`
- `内へ星を沈める力`
- `未来の星図を見る力`
- `地上の手触りを読む力`

## 7. 設問仕様

### 7.1 設問数

MVPは16問。各軸4問ずつ。

理由:

- 16タイプ診断らしさが出る
- 2分程度で完了できる
- 各軸に最低限の安定性を持たせられる
- 16という数字が世界観と相性がよい

### 7.2 設問トーン

悪い例:

- `人と話すのが好きですか？`
- `計画的ですか？`
- `論理と感情のどちらを重視しますか？`

良い例:

- `夜の図書館で未知の魔導書を見つけた。あなたはまず...`
- `星盤が示す道が二つに割れた。あなたが信じるのは...`

### 8.3 結果詳細文

#### INTJ: 星架 / 星図の建築家

概要:

遠くの未来を見据え、複雑な物事を設計図に変えるタイプ。感情に流されるより、構造・戦略・長期的な勝ち筋を重視する。孤高に見えても、内側では理想の完成形を静かに燃やしている。

強み:

- 長期戦略
- 本質を抜き出す力
- 独自の美学

注意点:

- 周囲がまだ理解していない段階で結論を急ぎやすい
- 完璧な設計にこだわりすぎて共有が遅れる

相性:

- ENFP: 新しい風を入れてくれる
- ENTP: 計画に刺激的な仮説を与える

アドバイス:

`完成図だけでなく、途中の星明かりも少し見せて。`

#### INTP: 碧理 / 蒼き論理術師

概要:

世界の裏側にある法則を探すタイプ。疑問を放置できず、複雑な概念を分解し、自分なりの理論へ組み替える。興味が灯った瞬間の集中力は、星盤の奥まで届く。

強み:

- 分析力
- 抽象思考
- 独創的な仮説

注意点:

- 考え続けて実行が後回しになる
- 感情的な合図を見落とすことがある

相性:

- ENTJ: 理論を現実に動かしてくれる
- ENFJ: 人との接続を助けてくれる

アドバイス:

`その美しい仮説を、一度だけ地上に降ろしてみて。`

#### ENTJ: 王星 / 王冠の指揮官

概要:

目的を定め、人と資源を動かし、結果へ向かって進むタイプ。判断が早く、混乱した場でも方向を示せる。強い意志の裏には「もっと良くできるはず」という高い基準がある。

強み:

- 決断力
- 組織化
- 目標達成

注意点:

- 速度が速すぎて周囲の感情が追いつかない
- 弱さや迷いを無駄だと切り捨てやすい

相性:

- INTP: 鋭い分析で戦略を磨く
- INFP: 目的に意味と温度を与える

アドバイス:

`進軍の前に、仲間の呼吸も一度だけ聞いて。`

#### ENTP: 月音子 / 蒼星の発明家

概要:

ひらめきと議論で世界を揺らすタイプ。常識を固定された壁ではなく、遊びながら壊せる仕掛けとして見る。新しい発想を生み、場に知的な火花を散らすムードメーカー。

強み:

- 発想力
- 即興力
- 議論と検証

注意点:

- 面白さを優先して継続が苦手になる
- 相手が大切にしている前提まで壊してしまう

相性:

- INTJ: 発想を設計へ変えてくれる
- INFJ: アイデアに深い意味を与える

アドバイス:

`遊び心に、最後のひと結びだけ添えて。`

#### INFJ: 深月 / 静謐の予言者

概要:

## 9. UI仕様

### 9.1 全体レイアウト

モバイルファースト。画像が1024 x 1536の縦長比率なので、結果カードは `2:3` に近い比率で設計する。

推奨ブレイクポイント:

- Mobile: `360px` から `767px`
- Tablet: `768px` から `1023px`
- Desktop: `1024px` 以上

最大幅:

- 診断UI: `min(100%, 480px)`
- 結果カード: `min(92vw, 420px)`、アスペクト比 `2 / 3`

### 9.2 背景

開始画面:

- `mbtishindancardura.png` を暗く敷く、または中央カードとして配置
- 背景には星粒の微細な発光
- 画面端は黒藍へ落とす

質問画面:

- 背景は深層夜空
- 中央に質問カード
- カードの枠は金線、内側は濃紺グラデーション

結果画面:

- カードが主役
- 周囲には余白を取り、UI操作ボタンはカード下に置く

### 9.3 コンポーネント

#### PrimaryButton

用途:

- 診断開始
- 回答選択
- 結果保存

見た目:

- 背景: 濃紺からサファイア
- 枠: 金線
- 文字: 月白
- hover/active: 宝石の発光
- 高さ: 最低 `48px`

#### ChoiceButton

仕様:

- 二択は縦並び
- タップ領域は十分広くする
- 選択肢文は2行までを基本にする
- 長文でもボタン外にはみ出さない

#### ProgressStars

仕様:

- 16個の小さな星または宝石
- 回答済みが金/青に発光
- 未回答は暗い青灰

#### ResultCard

仕様:

- 2:3比率
- 上部にタイプコード
- 中央にキャラクター/シンボル
- 下部にカード名、称号、引用
- 装飾過多でも文字可読性を最優先

### 9.4 フォント

推奨:

- 英字タイプコード: `Cinzel`, `Cormorant Garamond`, `Times New Roman`
- 日本語見出し: `Shippori Mincho`, `Noto Serif JP`
- 本文/UI: `Noto Sans JP`, `Hiragino Sans`

文字設計:

- タイプコードは大きく、文字間を少し広げる
- 日本語の引用文は明朝系でカード感を出す
- ボタン文は可読性優先でゴシック系

### 9.5 アニメーション

推奨:

- カードめくり: `transform: rotateY`
- 宝石発光: `box-shadow` と `filter`
- 星盤回転: ゆっくり `rotate`
- 結果出現: 透明度 + 少し上昇

制約:

- 派手にしすぎない
- 低スペック端末でも重くしない
- `prefers-reduced-motion` ではアニメーションを弱める

## 10. 画像アセット利用仕様

### 10.1 `mbtishindancardura.png`

推奨用途:

- 開始画面の中央カード
- 質問カードの裏面
- 集計中のカード
- 結果カードの背景テンプレート

加工:

- そのままだと装飾密度が高いので、テキストを載せる場合は暗い半透明レイヤーを重ねる
- 中央の宝石/星盤を活かす
- 端の金装飾を切らない

### 10.2 `mbtishindan.jpg`

推奨用途:

- ENTP結果サンプル
- 結果カードのトーン参考
- デザイン検証用の基準画像

加工:

- ENTP専用カードとして使う場合、文字の差し替えは画像編集ではなく、HTML/CSSでカードUIを再構成する方が安全
- 16タイプすべてを作る場合、この画像だけを流用するとタイプ差が弱くなるため、中央キャラ/シンボルの差し替えが必要

## 13. 非機能要件

パフォーマンス:

- 初回表示は3秒以内を目標
- 画像は適切に圧縮する
- アニメーションはCSS中心

アクセシビリティ:

- ボタンはキーボード操作可能
- 文字と背景のコントラストを確保
- タップ領域は44px以上
- アニメーション軽減設定に対応

可読性:

- スマホ幅でも選択肢文がはみ出さない
- 結果カードの文字は装飾に埋もれない
- 長い結果説明はカード外の詳細欄に置く

## 16. テスト計画

### 16.1 ロジックテスト

- 全回答をE寄りにしたら先頭がEになる
- 全回答をI寄りにしたら先頭がIになる
- S/N、T/F、J/Pも同様に判定される
- 同点時の挙動が仕様通り
- 16タイプすべてが到達可能

### 16.2 UIテスト

- 360px幅で開始画面が崩れない
- 選択肢が長文でもボタン内に収まる
- 16問目回答後に結果画面へ遷移する
- 戻るボタンで前問へ戻れる
- 再診断で状態が初期化される

### 16.3 ビジュアル確認

- カード裏面画像が暗すぎず、装飾が見える
- 結果カードの文字が金装飾に埋もれない
- 背景、カード、ボタンが一体の世界観になっている
- スクリーンショット時にタイプ名、称号、引用文が読める

## 17. 今後の拡張案

### 17.1 名前入力によるパーソナライズ

開始時に任意で名前を入力すると、結果カードに `〇〇の星術カード` と表示する。

注意:

- 入力必須にすると離脱が増えるため任意にする
- 名前はローカルのみで扱う

### 17.2 レア度演出

結果にレア度を付ける。

例:

- `Common`
- `Rare`
- `Epic`
- `Legendary`

ただしタイプに上下関係が出ないよう、レア度はランダム演出または日替わり演出にする。

### 17.3 相性診断

結果画面から `友達と相性を見る` を追加する。2人のタイプを選ぶと、関係性をカード風に表示する。

### 17.4 16キャラクター化

各タイプに専用キャラクター画像を用意する。現在の画像はENTPの基準として使い、他タイプは髪色、衣装、持ち物、背景天体で差別化する。

例:

- INTJ: 星図、黒金、塔
- INFP: 夢晶、薄紫、羽根ペン
- ESTP: 稲妻、短剣、流星
- ISFJ: 月灯、白金、鍵

## 18. 実装時の優先順位

1. 設問データと結果データを作る
